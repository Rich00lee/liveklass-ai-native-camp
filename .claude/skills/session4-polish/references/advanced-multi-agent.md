## EXPLAIN

### 멀티 에이전트 스킬이란?

Session 3에서 만든 스킬은 **Claude 1명이 순서대로 처리**하는 구조였다.
멀티 에이전트 스킬은 **여러 명의 Claude가 동시에 각자 맡은 일을 하는** 구조다.

**비유: 카페 운영**

```
단일 스킬 (1인 카페)
─────────────────────────
바리스타 1명이 순서대로:
  주문 받기 → 커피 만들기 → 서빙 → 다음 주문

멀티 에이전트 스킬 (팀 카페)
─────────────────────────
매니저가 지시하면:
  바리스타A: 커피 만들기  ─┐
  바리스타B: 디저트 준비  ─┤── 동시에 진행!
  바리스타C: 포장 준비    ─┘
  매니저: 다 끝나면 검수 → 손님에게 전달
```

### 핵심 패턴: 확산 → 수렴

멀티 에이전트 스킬의 뼈대는 거의 다 이 구조다:

```
입력 (분석할 데이터)
  │
  ▼
확산(Diverge) ──→ 여러 전문가가 각자 분석 (동시에)
  │                 전문가A: 이 관점에서 분석
  │                 전문가B: 저 관점에서 분석
  │                 전문가C: 또 다른 관점에서 분석
  │
  ▼
수렴(Converge) ─→ 검증자가 결과를 하나로 합침
  │
  ▼
사용자에게 통합 결과 제시
```

**실제 예시: /my-session-wrap 스킬**

```
오늘 작업 내용 (git 변경사항)
  │
  ├── doc-updater: 문서 업데이트 필요한 곳 찾기     ─┐
  ├── automation-scout: 자동화할 수 있는 패턴 찾기   ─┤ 동시 실행
  ├── learning-extractor: 오늘 배운 것 정리          ─┤ (Phase 1)
  └── followup-suggester: 다음에 할 일 제안          ─┘
                                                      │
                    duplicate-checker: 중복 제거       │ 순차 실행
                                                      │ (Phase 2)
                                                      ▼
                              통합 리포트 → 사용자 선택
```

### 병렬 vs 순차, 어떻게 구분할까?

에이전트를 설계할 때 가장 중요한 질문 하나:

> **"이 작업이 다른 작업 결과 없이 독립적으로 가능한가?"**

| 질문 | 답변 | 실행 방식 |
|------|------|----------|
| "문서 업데이트 찾기"는 "자동화 패턴 찾기" 결과가 필요한가? | **아니요** (각자 다른 영역 분석) | 병렬 |
| "중복 검증"은 위 4개 결과가 필요한가? | **네** (결과가 있어야 비교 가능) | 순차 |

```
독립적인 작업들 ──→ 병렬로 동시 실행 (속도 향상)
앞 결과에 의존하는 작업 ──→ 순차로 뒤에 실행 (정확성 보장)
```

### 에이전트 역할 분리 원칙

좋은 멀티 에이전트 스킬의 각 에이전트는 이 4가지를 갖고 있다:

```
1. 역할 (한 줄) ──── "이 에이전트는 ○○만 한다"
2. 도구 (최소한) ─── 읽기 도구만 (분석만 하고, 수정은 메인이)
3. 분석 대상 ────── 구체적인 체크리스트
4. 출력 형식 ────── 마크다운 템플릿 고정
```

**나쁜 예**: "전체 코드를 분석하고 문서도 수정하고 테스트도 돌려줘"
**좋은 예**: "변경된 파일 목록을 받아서, 문서 업데이트가 필요한 곳만 찾아줘"

→ **한 에이전트 = 한 가지 일** (단일 책임)

### 비용 최적화 팁

모든 에이전트가 똑같이 똑똑할 필요는 없다:

| 역할 | 추천 모델 | 이유 |
|------|----------|------|
| 분석/생성 (깊이 필요) | **Sonnet** | 패턴 인식, 통찰력 추출에 강함 |
| 검증/비교 (단순 매칭) | **Haiku** | 빠르고 저렴, 단순 비교에 충분 |

→ **"생성은 똑똑한 모델, 검증은 가벼운 모델"**

### 4단계 설계 프로세스

멀티 에이전트 스킬을 만들고 싶을 때 이 순서를 따른다:

**Step 1: 수동으로 먼저 해보기**
> "이 작업을 내가 직접 한다면 어떤 단계를 거칠까?"

예) "CS 문의 분석 스킬"을 만든다면:
- 문의 내용 읽기 → 카테고리 분류 → 기존 답변 검색 → 초안 작성 → 톤 검수

**Step 2: 독립/의존 관계 그리기**
```
문의 읽기 ──→ 카테고리 분류 (독립)
         ──→ 기존 답변 검색 (독립)
                    ↓
              답변 초안 작성 (위 2개 결과 필요 → 순차)
                    ↓
              톤 검수 (초안 결과 필요 → 순차)
```

**Step 3: 에이전트로 뽑을지 판단**
- 분석이 복잡하고 도구 호출이 여러 번 필요? → 에이전트로 분리
- 단순한 변환이나 1-2줄 처리? → 메인 스킬에서 직접

**Step 4: 출력 형식 통일**
- 각 에이전트의 결과 형식을 미리 정한다
- 나중에 통합할 때 형식이 제각각이면 합치기 어려움

## EXECUTE

직접 해보기 — 내가 만든 Skill을 멀티 에이전트로 확장하는 상상 실험:

Claude Code에 입력:
```
내 Skill [이름]의 처리 과정을 단계별로 나눠줘.
각 단계가 독립적인지, 앞 단계 결과에 의존하는지 표시해줘.
독립적인 단계들을 병렬로 묶으면 어떤 구조가 되는지 다이어그램으로 보여줘.
```

→ 이 결과를 보고 "진짜 에이전트로 나눌 가치가 있는지" 판단해보세요.
→ 단계가 2개 이하로 나뉘면? 멀티 에이전트가 아니라 단일 스킬로 충분합니다.

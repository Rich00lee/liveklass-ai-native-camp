# LiveKlass AI Native Camp 1기 회고

> 2026.03.03 ~ 03.13 | 5세션, 2주, 9명 (일반 7 + 앰버서더 2)
> 원본 커리큘럼: [ai-native-camp/camp-1](https://github.com/ai-native-camp/camp-1) (장준, 김성중, 정구봉)

---

## 1. 왜 시작했나

2026년 초, 사내에 AI 에이전트를 배포하려 했다. Claude Code를 설치해주고, 스킬을 만들어 공유하면 팀이 바뀔 거라 생각했다. 결과는 뻔했다. 아무도 안 썼다.

도구를 던져준다고 사람이 바뀌지 않는다. **멘탈모델(mental model)이 바뀌어야 한다.** "내 업무에서 반복되는 30분을 찾아내고, 그걸 명령어 하나로 줄일 수 있다"는 감각. 그걸 경험으로 심어줘야 했다.

ai-native-camp/camp-1을 직접 체험한 뒤 확신이 생겼다. 비개발자 45명 중 80%가 Day 1에 자기 업무 도구를 만들어냈다. 우리 팀도 할 수 있다. 단, 우리 업무 맥락에 맞게 커스텀해야 했다.

---

## 2. 어떻게 설계했나

### Skills as Curriculum

슬라이드를 만들지 않았다. 커리큘럼 자체가 Claude Code Skill이다. `/session1-onboarding`을 실행하면 Claude가 직접 가르친다. 진행자는 옆에서 코칭만 한다.

이 방식의 장점은 명확했다. 참가자가 "배우는 도구"와 "배우는 환경"이 동일하다. 슬라이드를 보고 따라치는 게 아니라, 터미널 안에서 배우고 터미널 안에서 만든다.

### STOP PROTOCOL

AI 교사의 가장 큰 문제는 멈추지 않는 것이다. 설명하고, 질문하고, 답을 기다리고, 다음으로 넘어가는 걸 한 턴에 다 해버린다. 학습자가 실행할 틈이 없다.

STOP PROTOCOL은 이걸 강제로 끊는다. 각 블록은 Phase A(설명 + 실행 안내)에서 반드시 STOP한다. 학습자가 직접 실행한 뒤 다시 말을 걸면 Phase B(퀴즈 + 다음 블록 이동)가 시작된다. 2턴(two-turn) 구조. STOP 전에 질문하거나 다음 블록을 언급하는 것은 명시적으로 금지했다. 명시하지 않으면 AI가 위반한다.

### 결과물 중심

5세션의 목표는 하나였다. **Session 5에서 전사 앞에 서서 "내가 만든 자동화"를 시연하는 것.** 발표일이 정해져 있으니 역산해서 설계했다. 학습 4세션 + 발표 1세션.

## 4. 무엇이 바뀌었나

캠프는 3/13에 끝났다. 하지만 바뀐 것들은 캠프 밖에서 일어났다.

- **캠프 이후에도 계속 만드는 사람들**: Terry는 캠프 종료 2일 뒤(3/15)에도 CX 어시스턴트를 개선하고 있었다. memory 파일 자동 로드 구조 추가, 토큰 절감 검색(약 60% 절감), CLAUDE.md에 CX 자동화 섹션 추가. "이 어시스턴트가 CX 셀 공용으로 쓰이게 된다면 팀 단위 정책 논의가 필요하다"는 인사이트까지 도달했다.
- **멘탈모델의 전환**: Vivi는 캠프 다음 날(3/14) "산출물 퀄리티가 자꾸 낮아지는 것 같다"며 외부 학습 자료를 찾아 채널에 공유했다. "모든 병목은 결국 제 자신이라는 점에 웃음이 나오는 건, 옛날이었으면 낙담했을 텐데 이제는 '고쳐보지 뭐'라는 마인드셋이 되는 건 정말 재미난 경험"이라는 소감을 남겼다.
- **보안 의식의 자발적 작동**: Ethan은 Session 5 발표 직후 "고객사, 회사 정보가 발표자료에 포함되어 있어서 보안상 비공개로 돌려야 하지 않을까"라고 제기했다. showcase 레포가 private으로 전환되는 계기가 됐다.
- **LinkedIn 공유**: Rich가 캠프 과정에서 얻은 인사이트를 LinkedIn "삽질 시리즈"로 발행했다. 캠프 경험이 외부 콘텐츠로 이어진 사례다.

도구를 배운 것이 아니라 "내 업무를 들여다보는 습관"이 생겼다. 캠프가 끝나도 그 습관은 멈추지 않았다.

## 5. 커뮤니티가 만든 가치

캠프 기간 중 `#lk-ai-camp` 채널에서 참가자들이 자발적으로 만들어낸 것들이 있다. 커리큘럼에 없던 것들이다.

- **이보배 (Vivi)**: Session 2 리캡 노트를 채널에 정리해 공유. 캠프 참가자들의 Notion 멤버 권한 확보를 직접 주도. Claude Code 활용 품질 개선을 위한 외부 학습 자료(영상 + 가이드 노션)를 찾아 공유. 발표회 사진·영상 아카이빙까지 담당했다.
- **최재훈 (Terry)**: Session 1 과제(도메인 CNAME 가이드)에서 출발해, 주말에 혼자 React 앱을 만들어 GitHub에 push했다 (DNS 설정 도우미). 캠프 종료 후에도 CX 어시스턴트를 지속 개선하며, "팀 공용 어시스턴트로 확장하려면 memory 파일을 팀 공용 GitHub 레포에서 관리해야 한다"는 아키텍처 인사이트를 공유했다.
- **이새봄 (Bom)**: 집 PC → 회사 PC 작업 연결을 위해 GitHub 레포 + session-notes.md 조합을 직접 고안. Jay의 PR 제출을 돕기 위해 구체적인 커맨드를 채널에 공유했다. 캠프 사진도 매 세션마다 기록했다.
- **김영호 (Ian)**: "비개발자가 AI로 직접 처리하게 되면, 개발자의 역할은 어떻게 달라져야 할까?"라는 질문을 Session 1 과제로 던지고, 개발자 역할의 3가지 축(시스템 설계자, 도구의 도구를 만드는 사람, 품질의 최종 방어선)을 정리했다. Arin이 보드게임 채널에서도 AI를 활용하는 것을 발견하고 채널에 공유하기도 했다.
- **김성원 (Ethan)**: 발표 자료 내 고객사·회사 정보 노출 리스크를 즉시 제기하여 showcase 레포 보안 전환의 계기를 만들었다.
- **송상수 (Eddie)**: 첫 세션 소감으로 "어떤 도구를 쓰더라도 AI와 함께 일한다는 것을 전제로 일하는 방식 자체를 바꾸는 것이 중요하다"는 메시지를 남기며 캠프의 방향성을 환기했다.

9명이라는 작은 규모였지만, 서로의 과제를 돕고, 자료를 공유하고, 보안 이슈를 짚어주는 흐름이 자연스럽게 만들어졌다.

## 6. 배운 점과 한계

**효과가 있었던 것:**
- STOP PROTOCOL은 AI 교사의 폭주를 확실히 막았다. 금지 사항을 명시적으로 적지 않으면 AI는 반드시 위반한다는 것을 확인
- 발표일 고정(Session 5)이 가장 강력한 동기부여. 역산 설계가 작동했다
- Skills as Curriculum 방식은 진행자 의존도를 크게 낮췄다. 9명이 동시에 각자 속도로 진행 가능

**솔직한 한계:**
- **Figma API 제약**: Bom의 디자인 시스템 자동화가 원래 범위를 달성하지 못했다. MCP로 연결할 수 있는 도구와 없는 도구의 경계를 사전에 더 명확히 안내했어야 했다
- **토큰(token) 비용**: Team Standard 시트 기준으로도 참가자 9명이 2주간 적지 않은 토큰을 소모했다. 학습 과정에서의 시행착오가 많을수록 비용이 늘어난다. 조직 차원의 비용 정책이 선행되어야 한다
- **온/오프 병행의 어려움**: Session 2~4는 온라인 참여를 허용했는데, 온라인 참가자의 트러블슈팅 지원이 오프라인 대비 확연히 어려웠다
- **앰버서더 역할 정의 부족**: 개발자 앰버서더 2명의 역할이 "기술 지원"으로만 정의되어 있었는데, 실제로는 커뮤니티 활성화, 과제 리뷰 등 더 넓은 역할이 필요했다

## 7. 재현하려면

이 캠프의 커리큘럼은 오픈소스(open source)다.

### 원본 설치

```bash
# ai-native-camp/camp-1 원본 (장준, 김성중, 정구봉)
git clone https://github.com/ai-native-camp/camp-1.git
cd camp-1
# Claude Code에서 /session1 등 스킬 실행
```

### LiveKlass 커스텀 버전

```bash
git clone https://github.com/liveklass/liveklass-ai-native-camp.git
cd liveklass-ai-native-camp
# .claude/skills/ 아래 session1~5 스킬 확인
# 참가자 산출물: lk-ai-camp-showcase/submissions/
```

### 커스텀 가이드

자기 조직에 맞게 커스텀하려면:

1. `camp-1`을 fork(복제)한다
2. `references/*.md`의 실습 예시를 자기 업무 맥락으로 교체한다 (CS 문의 → 자기 팀의 반복 업무)
3. STOP PROTOCOL과 금지 사항 구조는 그대로 유지한다 — 이걸 빼면 AI 교사가 제어 불가
4. Session 5(발표일)를 먼저 잡고 역산한다
5. 앰버서더(개발자)를 최소 1명 확보한다 — MCP 설치 트러블슈팅에 필수
